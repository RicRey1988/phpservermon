# PHP Server Monitor 4.3.2-hs — Native HOPE UI Migration Design

## Goal

Remove the custom `src/templates/default/static/css/hs-monitor.css` stylesheet and migrate the application shell and module templates to the native HOPE UI and Bootstrap 5 component contracts already bundled by the project. The result must fix the navbar theme control, search field, sidebar collapse direction, duplicated sidebar user session block, and mobile sidebar behavior while preserving all existing application features.

## Approved scope

- Delete `src/templates/default/static/css/hs-monitor.css` completely.
- Remove every runtime and service-worker reference to that stylesheet.
- Keep only the official bundled HOPE UI stylesheets: `hope-ui.min.css`, `dark.min.css`, and `customizer.min.css`.
- Replace presentation classes that depended on `hs-monitor.css` with native HOPE UI/Bootstrap 5 cards, grids, forms, navigation, badges, alerts, lists, avatars, spacing, display, positioning, overflow, and responsive utilities.
- Replace styling-only custom classes with native classes. Where JavaScript needs stable hooks, use `data-*` attributes rather than CSS classes.
- Preserve PHP 8.5 compatibility, Twig rendering, accessibility, current data flows, PWA behavior, notifications, charts, updater, profile media, branding, and appearance persistence.

## Application shell

### Sidebar

The sidebar will use HOPE UI's native structure:

- `aside.sidebar.sidebar-default.sidebar-white.sidebar-base.navs-rounded-all`
- a standard `sidebar-header` containing the brand and native `sidebar-toggle`
- a `sidebar-body` and `sidebar-list` containing the primary navigation
- `sidebar-mini` as the single collapsed/hidden state managed by HOPE UI

The desktop toggle will sit on the outside edge of the expanded sidebar and its icon will rotate automatically when `sidebar-mini` is active, matching the upstream template. The toggle will not use generic button classes that override HOPE UI's sidebar control geometry.

At responsive widths, the sidebar starts off-canvas through the native `sidebar-mini` state. The navbar toggle removes that state to open the sidebar and restores it to close. `aria-expanded`, button labels, Escape handling, outside-click handling, and closing after navigation will be synchronized in `app-shell.js`. The obsolete `body.sidebar-open` contract will not exist.

### Sidebar menu content

Only primary application navigation remains in the sidebar. The duplicated lower session section containing the current user label, Profile, and Logout is removed from `main/menu.tpl.html`; Profile and Logout remain in the authenticated navbar user dropdown.

### Navbar search

The search field will follow the upstream HOPE UI `input-group search-input` markup, including a joined search icon and input with native border, background, focus, and dark-mode behavior. Custom width and color rules are removed. Responsive visibility will use Bootstrap display utilities.

### Theme control and navbar icons

The theme control will occupy the same native icon-button box as settings and notifications. Only one theme icon is presentationally visible at a time, with visibility controlled through the standard `d-none`/display utility classes from JavaScript rather than custom CSS selectors. The button label and title will describe the next action.

All top-bar SVGs will inherit current color, use consistent HOPE UI icon dimensions, and remain vertically centered in light and dark schemes. No per-theme custom color override will be required.

## Module migration strategy

Each current custom presentation contract will be replaced by native composition rather than recreated in another stylesheet.

- Application layout: Bootstrap containers, rows, columns, flex, grid, gap, width, overflow, position, and spacing utilities.
- Cards and collections: HOPE UI/Bootstrap `card`, `card-header`, `card-body`, `list-group`, responsive `row-cols-*`, and `h-100`.
- Forms: `form-label`, `form-control`, `form-select`, `input-group`, `form-check`, `form-switch`, validation feedback, and responsive grid columns.
- Configuration and detail navigation: native `nav`, `nav-pills`, `tab-content`, sticky utilities where supported, and responsive overflow utilities.
- Status presentation: native semantic `bg-*`, `text-*`, `border-*`, badges, alerts, icon containers, and cards. JavaScript will swap complete Bootstrap class sets from a controlled allow-list.
- Authentication: native responsive rows/columns and cards using the official HOPE UI authentication visual assets.
- Images and avatars: native avatar utilities where available plus intrinsic `width`/`height` attributes and Bootstrap object-fit/rounded utilities; no custom sizing stylesheet.
- Timelines, server facts, diagnostics, installers, and empty states: cards, list groups, borders, badges, progress/step indicators, and responsive grid/flex utilities.
- Charts and preformatted output: Bootstrap ratio/overflow utilities and safe element attributes; JavaScript-owned dimensions only when a chart library requires them.

Semantic classes that currently exist only for styling will be removed. Stable behavior selectors such as server refresh, search, appearance settings, uploads, and status updates will move to named `data-*` hooks.

## JavaScript contracts

`app-shell.js` will own the small amount of application-specific interaction not supplied by HOPE UI:

1. synchronize desktop and mobile sidebar state with `sidebar-mini`;
2. update `aria-expanded` and accessible labels on every sidebar toggle;
3. close the mobile sidebar on Escape, outside click, or primary navigation;
4. toggle the theme icon's Bootstrap display utilities after appearance changes;
5. retain card search, password visibility, modal, dropzone, and conditional-form behavior through `data-*` hooks;
6. avoid adding inline presentation styles except values required by chart libraries or image previews.

The bundled `hope-ui.js` remains the primary source for native sidebar behavior. The application layer will complement it only where the upstream implementation does not provide accessible mobile state synchronization.

## Dark and light modes

HOPE UI's official light and dark stylesheets remain authoritative. Templates will use theme-aware Bootstrap variables and semantic utilities instead of literal colors. All migrated pages must be visually usable in both modes, including navbar controls, forms, dropdowns, cards, alerts, status indicators, offcanvas/customizer content, and authentication pages.

## PWA and cache behavior

`service-worker.js` will remove `hs-monitor.css` from `STATIC_ASSETS` and rotate its static cache revision so already installed clients stop serving the deleted file. The HTML template will also remove the stylesheet link. No private or dynamic response will be added to the cache.

## Compatibility and error handling

- No database schema changes are required.
- Existing Twig data contracts remain stable unless a field exists solely to render the removed sidebar user block.
- Failure of an optional image or chart asset must not prevent navigation or primary content rendering.
- Sidebar and theme controls must remain usable when persistent appearance saving fails; the existing toast/error recovery flow remains intact.
- JavaScript must safely no-op on pages that omit a relevant control.

## Verification

Automated contracts will require:

- no `hs-monitor.css` file;
- no HTML or service-worker reference to `hs-monitor.css`;
- no obsolete `body.sidebar-open` selector or script contract;
- no duplicated `.sidebar-user`/Profile/Logout block in the sidebar;
- native HOPE UI sidebar markup and `sidebar-mini` behavior;
- Bootstrap utility-driven theme icon visibility;
- native search input-group markup;
- no Bootstrap 4 attributes or legacy directional utility classes;
- all templates to render without missing Twig data;
- service-worker cache revision to be updated.

Verification will include PHP syntax checks, the complete PHPUnit suite, static searches for removed contracts, and browser checks at desktop, tablet, and mobile widths in both light and dark modes. Authenticated checks will cover every primary menu destination, sidebar open/close behavior, search, appearance persistence, dropdowns, status refresh, charts, forms, media uploads, notifications, updater, and logout.

## Delivery

Implementation will be committed in reviewable stages, deployed to the test VPS while preserving `config.php`, uploads, logs, database content, and server-specific permissions, then validated over HTTPS. After successful repository and VPS verification, the final commits will be published to `main`, tagged `v4.3.2-hs`, and included in the GitHub release notes.
