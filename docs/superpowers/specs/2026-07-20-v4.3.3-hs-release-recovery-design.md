# v4.3.3-hs Release Recovery and VPS Deployment Design

## Context

The production installation at `monitoreo.hostingsupremo.org` is running
`4.3.2-hs`. Its deployed navbar, sidebar menu, and application-shell JavaScript
still contain the UI behavior shown in the reported screenshots.

The `main` branch already contains the intended Hope UI corrections and declares
`PSM_VERSION` as `4.3.3-hs`, but the corresponding tag and GitHub release do not
exist. The release push failed before jobs started because three GitHub Actions
workflow files are syntactically invalid:

- `.github/workflows/release-hs.yml` contains a plain YAML scalar with `: ` in
  the release condition.
- `.github/workflows/php85.yml` contains an unindented shell heredoc terminator.
- `.github/workflows/vps-ops-chatops.yml` contains the same heredoc indentation
  error.

The historical updater URL uses `RicRey1988/phpservermon`; GitHub redirects that
renamed repository path to
`RicRey1988/phpservermon-Redesigned-by-hostingsupremo`, so the current repository
remains the canonical source and release channel.

## Objective

Recover and publish the signed `v4.3.3-hs` release, deploy it safely to the VPS,
and demonstrate that all four requested navigation corrections work in desktop
and responsive layouts under light and dark themes.

## Scope

The implementation will:

1. Repair the invalid workflow syntax without changing unrelated automation
   behavior.
2. Add a regression check that parses all active GitHub Actions workflow files
   with GitHub Actions-compatible rules.
3. Re-run the full PHP, static-analysis, JavaScript, packaging, updater, and UI
   contract checks.
4. Correct any remaining defect found while rendering the existing Hope UI
   implementation; no unrelated visual redesign is included.
5. Publish an immutable `v4.3.3-hs` tag and signed release assets.
6. Back up and update the test VPS, then perform application and visual health
   verification.

## UI Acceptance Criteria

The release is acceptable only when all of the following are true:

- The quick light/dark control has the same centered box and baseline as the
  settings and notification controls. Its moon and sun glyphs remain visible
  against their respective themes.
- The search input uses the native Hope UI `input-group search-input` structure,
  with a theme-safe surface, border, icon, placeholder, and focus state.
- The desktop sidebar collapse arrow uses the native Hope UI sidebar header
  position and points toward the side to which the sidebar collapses. Its state
  remains synchronized after expanding and collapsing.
- The duplicate sidebar session block containing the username, Profile, and
  Sign out is absent; these actions remain available in the navbar account menu.
- At mobile widths, the navbar menu control opens the sidebar. The sidebar closes
  after navigation, outside click, or Escape, and `aria-expanded` reflects the
  actual state.
- No legacy custom `hs-monitor.css` stylesheet or removed presentation hook is
  reintroduced.

## Implementation Design

### Workflow recovery

The release condition will be expressed as an explicit GitHub Actions expression
so YAML cannot interpret the colon in `release: v` as a mapping separator. Both
shell heredoc terminators will be indented as YAML block content; YAML removes
that structural indentation before Bash evaluates the script, leaving the
terminator at column zero in the executed program.

A repository-level workflow syntax test will cover every `.yml` and `.yaml` file
under `.github/workflows`. It must fail on the current three defects and pass only
after all three are repaired.

### Application verification

Existing UI contract tests will verify the Hope UI markup and behavior hooks.
The responsive audit will cover widths 360, 390, 768, 1024, 1366, and 1600 in
light and dark modes. The audit will check sidebar interaction, theme switching,
settings access, horizontal overflow, and the four reported regressions.

If the rendered application exposes a mismatch not caught by current tests, a
focused failing regression test will be added before the smallest corresponding
template or JavaScript change.

### Release publication

The recovery changes will be integrated into `main` with release identity
`v4.3.3-hs`. The release workflow will:

1. Verify source version and release notes.
2. Run Composer validation, platform requirements, dependency audit, PHPUnit,
   PHPStan, and JavaScript syntax checks.
3. Build the production ZIP and canonical SHA-256 manifest.
4. Sign the manifest with the existing repository release secret and verify it
   with the bundled public key.
5. Create an immutable annotated `v4.3.3-hs` tag.
6. Publish and re-read the ZIP, JSON manifest, and detached signature assets.

No deployment starts unless the tag, release, signature, manifest, and asset set
are complete and mutually consistent.

### VPS deployment

Before mutation, the deployment process will record the application version,
service states, application ownership, available disk space, and a timestamped
backup location. It will create recoverable backups of:

- the application tree, including private configuration and uploaded/runtime
  data;
- the MariaDB application database;
- the current Apache virtual-host configuration relevant to the application.

The signed release will then be applied through the supported updater path or an
equivalent verified staging deployment that preserves `src/config/config.php`,
logs, uploaded images, and runtime data. The deployed tree must retain the
existing `hostin:apache` ownership contract and required setgid directory modes.

Post-deployment commands will run database migrations and application health
checks. Apache configuration and the active states of Apache, PHP-FPM, and
MariaDB will be checked before browser acceptance testing.

### Rollback and failure handling

Failures are gated by phase:

- Invalid source, tests, signature, or assets stop before release publication.
- An incomplete release stops before VPS mutation.
- A failed backup stops deployment.
- A failed migration, health check, HTTP check, or authenticated browser smoke
  test triggers restoration from the timestamped application and database
  backups, followed by service and HTTP verification.

The immutable tag will never be force-moved. If publication partially succeeds,
the same tag may be completed only when it still points to the verified release
commit; otherwise the process stops.

## Verification Matrix

| Layer | Verification |
| --- | --- |
| Workflow syntax | All active GitHub Actions files parse successfully |
| PHP dependencies | `composer validate --strict`, platform requirements, audit |
| Application | Complete PHPUnit suite and PHPStan |
| JavaScript | Syntax checks for service worker and application runtimes |
| Packaging | Production install, ZIP content, manifest hash, signature verification |
| Release | Immutable tag plus exactly the ZIP, JSON, and JSON signature assets |
| VPS | Version, migration, health, ownership, services, Apache config, HTTPS |
| UI | Four reported fixes across desktop/mobile and light/dark themes |

## Completion Criteria

Work is complete when GitHub exposes `v4.3.3-hs` as a non-draft stable release
with all signed assets, the VPS reports `4.3.3-hs`, automated checks pass, and an
authenticated visual audit confirms every UI acceptance criterion without
horizontal overflow or responsive navigation failure.
